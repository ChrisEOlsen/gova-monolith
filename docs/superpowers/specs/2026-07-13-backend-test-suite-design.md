# Backend Test Suite Design

**Status:** Approved for planning
**Author:** Claude (Lead Architect), with Chris Olsen

## Context

`/build`'s Step 3 has always said "Skip TDD — there is no test suite." That was
true by omission, not by design choice: GOVA apps are scaffolded via MCP tools
(deterministic boilerplate) and then hand-customized by an implementer subagent
(Step 3 of the Golden Recipe). That customization step is where real,
non-generic logic accumulates — and it recurs. An app isn't built once and
left alone; `/build` runs again, or a subagent extends an existing handler
weeks later, with no automatic way to know it broke a feature three files
over. `docker compose logs` and manual click-through only catch what someone
thinks to check.

The auth scaffolds are the sharpest version of this problem: this session
found and fixed four real bugs in them (CSRF exemption gap on the mobile
bootstrap request, missing rate limiting on mobile login, missing timing-safe
comparison on mobile login, missing email case normalization) — all in code
generated by a tool that's supposed to be more reliable than hand-written
code. A regression suite over exactly this code is the strongest cost/benefit
case in the whole template.

## Goals

- Go-only backend tests (models, handlers) using the standard library
  `testing` + `net/http/httptest` — no new dependency.
- Test isolation from the live app: tests never touch `/data/app.db`.
- Auth scaffolds get tests covering the bug classes already found in this
  codebase, not just generic CRUD.
- Scaffolded code gets its test generated by the same MCP tool call that
  generates the code — consistent with the "MCP tool first" philosophy
  already governing everything else in this template.
- Hand-customized code (bespoke `create_handler`/`create_page` stubs, and any
  scaffolded handler a later task customizes) gets a test-writing step folded
  into the implementation plan, same as the code itself.

## Non-Goals

- **JS/frontend testing.** CLAUDE.md's Critical Constraint #4 bans Node/npm;
  every standard JS test runner needs Node. Out of scope — DOM logic and
  client-side wiring stay manually/browser-verified, as today.
- **A repo-wide constraint-enforcement test** (grepping for `innerHTML` or raw
  SQL outside `models/` and failing `go test` on a hit). Considered and
  explicitly not chosen for this pass — task-reviewer subagent review remains
  the enforcement mechanism for CLAUDE.md's Critical Constraints.
- **CI pipeline integration** (GitHub Actions, etc.). Out of scope — this is
  about the `/build` workflow's own verification loop, not a public CI story.

## Architecture

### Test database isolation

`src/app/db/db.go`'s `Open(path string)` already accepts an arbitrary path
(defaulting to `/data/app.db` only when empty) — no changes needed there.
Tests use a **temp file**, not SQLite `:memory:`. `db.Open` returns two
separate `*sql.DB` handles (`Write`, `Read`); a literal `:memory:` DSN without
`cache=shared` gives each handle its own private database, so a row written
via `Write` would be invisible to `Read` — a silent, confusing bug. A
`t.TempDir()`-based file sidesteps this entirely and exercises the same
WAL-mode code path production uses.

New file: **`src/app/db/testutil.go`** (hand-written infrastructure, no
scaffold tool applies — same category as `middleware/*.go`). Exposes one
helper, roughly:

```go
func OpenTest(t *testing.T, schema string) *DB {
	t.Helper()
	path := filepath.Join(t.TempDir(), "test.db")
	d, err := Open(path)
	if err != nil {
		t.Fatalf("open test db: %v", err)
	}
	if _, err := d.Write.Exec(schema); err != nil {
		t.Fatalf("apply schema: %v", err)
	}
	t.Cleanup(func() { d.Close() })
	return d
}
```

Generated test files call `db.OpenTest(t, schema)` with a `CREATE TABLE`
string derived from the same `fields` argument already passed to
`model.go.tmpl` — one source of truth for column names/types, so the model
struct and the test schema cannot drift apart.

### Test invocation

`go test ./...` runs **on-demand** — `docker compose exec app go test ./...`
— not baked into `entrypoint.sh` or the Dockerfile's `builder` stage.
Rationale: the `app` container live-rebuilds from the bind-mounted source on
every restart (`entrypoint.sh` runs `go build` fresh each time); a build-time
test step would test a stale image snapshot, and gating `entrypoint.sh` on
tests passing would prevent a developer from starting a known-broken app to
debug it interactively. `CGO_ENABLED=1` and `gcc` are already present in the
`app`/`builder` image (needed today for the `go-sqlite3` driver), so no image
changes are required to make `go test` work.

## Components

1. **`src/app/db/testutil.go`** — new, hand-written. `OpenTest` helper described above.

2. **New template files in `src/builder/templates/`:**
   - `model_test.go.tmpl` — CRUD roundtrip (Create → FindByID → GetAll → Delete) against a temp db, schema derived from `fields`.
   - `list_handler_test.go.tmpl` — `httptest.NewRecorder()`/`httptest.NewRequest()` against the generated JSON handler, asserting status code and response shape.
   - `auth_test.go.tmpl` — login success, login failure (wrong password via the real bcrypt path, not the dummy-hash path), rate-limit trip after 5 failed attempts within the window, CSRF token rejection on a missing/mismatched `X-CSRF-Token` header.
   - `register_test.go.tmpl` — registration success, duplicate-email rejection, email lowercasing on create.
   - `mobile_auth_test.go.tmpl` — `login_token` issuance, Bearer-token validation on a protected endpoint, rate limiting and timing-safe comparison on the mobile path specifically (the exact class of bug fixed this session).

3. **`src/builder/main.go` changes:** `handleScaffoldList`, `handleCreateModel`, `handleScaffoldAuth`, `handleScaffoldRegistration`, and `handleScaffoldMobileAuth` each get one more entry in their `renderToFile` spec list, pointing at the matching `_test.go.tmpl`, output alongside the code file it tests (`models/Project_test.go` next to `models/Project.go`, etc.) — standard Go convention, same package/directory.

4. **`gova-writing-plans` update:** the Task Structure template gains a step for tasks that customize a bespoke stub (`create_handler`/`create_page`) or hand-edit a scaffolded handler: "write a test for the custom behavior" — planned per-task like the code itself, since there's no generic test to auto-generate for hand-written logic.

5. **`build.md` Step 8 update:** add `docker compose exec app go test ./...` as a required check, alongside (not replacing) the existing `docker compose logs app` check.

6. **`gova-build-execution` update:** `implementer-prompt.md` and `task-reviewer-prompt.md` add `go test ./...` (or the specific package under test) to the verification/evidence expected in the implementer's report, alongside the restart+logs check already there.

7. **`TODO.md`:** the one-line placeholder gets replaced with a pointer to this spec and the implementation plan once written.

## Risks / Open Considerations

- **bcrypt cost factor in tests.** Real bcrypt hashing at production cost is slow enough to matter across many test cases. The plan should specify using a lower `bcrypt.Cost` in the test build path — needs a task to confirm whether the auth handler template already parameterizes cost or needs a small change to allow overriding it in tests without weakening production security.
- **Rate-limit tests are time-sensitive** (5 attempts / 15 min window, per CLAUDE.md). Tests need to drive the rate limiter through its actual attempts/window logic without sleeping real wall-clock minutes — likely by inserting rows into `rate_limits` directly via the temp db rather than making 5 real HTTP round trips against a real clock.
- **Existing apps** (already-scaffolded models/handlers from before this change) won't retroactively get tests — only future scaffold calls generate them. Out of scope to backfill.
- **Compile-coupling on the upgrade path.** `register_test.go.tmpl`/`mobile_auth_test.go.tmpl` reference `authTestSchema`/`loginBody` from `auth_test.go.tmpl`, which only exist if `scaffold_auth` ran after this change shipped. An app that scaffolded auth *before* upgrading its builder (so has `handlers/auth.go` but no `handlers/auth_test.go`), then later runs `scaffold_registration`/`scaffold_mobile_auth`, would emit a test referencing undefined identifiers — breaking `go test` compilation for the whole `handlers` package, not just skipping a test. Same backfill limitation as above, but worth knowing it fails loud (compile error) rather than quiet (missing test) for that specific upgrade path.
